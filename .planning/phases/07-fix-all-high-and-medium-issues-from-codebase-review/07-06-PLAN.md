**Issue 28 — SQL LIMIT in get_history/get_aggregates (store.py:183-193, 272-281):**

In `get_history()`, add a `max_rows: int = 10000` parameter and append `LIMIT ?` to the SQL query with `max_rows` as the parameter:
```python
async def get_history(self, category: str, metric_name: str, hours: float = 1.0, max_rows: int = 10000) -> list:
```
The SQL should end with `ORDER BY timestamp ASC LIMIT ?` and pass `max_rows` as the last bind parameter.

In `get_aggregates()`, similarly add `max_rows: int = 10000` and apply a LIMIT to the underlying query (or limit the result set after aggregation if the query is more complex).

Add a `count_errors(hours: float = 24.0) -> int` method to `MetricsStore`:
```python
async def count_errors(self, hours: float = 24.0) -> int:
    """Direct COUNT of error events within the given time window."""
    cutoff = (datetime.utcnow() - timedelta(hours=hours)).isoformat()
    async with aiosqlite.connect(self._db_path) as db:
        cursor = await db.execute(
            "SELECT COUNT(*) FROM events WHERE timestamp >= ? AND event_type IN ('execution_error', 'error')",
            (cutoff,),
        )
        row = await cursor.fetchone()
        return row[0] if row else 0
```
Adjust the event_type filter to match actual event types used in the codebase (check `record_event` calls for exact event type strings).

**Issue 30 — Duplicate route handler logic (dashboard.py):**

Extract the handler body logic into module-level or helper functions that both `create_monitoring_app()` and `register_monitoring_routes()` call. For example:

```python
async def _handle_health(state: Any) -> dict:
    """Shared health check logic."""
    pool_status = state.pool.get_status() if state.pool else {}
    return {"status": "ok", **pool_status}

async def _handle_metrics(state: Any) -> dict:
    """Shared metrics logic."""
    # ... existing metrics body ...
```

Then in both `create_monitoring_app()` and `register_monitoring_routes()`:
```python
async def health_handler(request: Request) -> JSONResponse:
    return JSONResponse(await _handle_health(state))
```

This way the business logic is defined once. The thin handler wrappers in each registration function just call the shared function and wrap in the appropriate response type.

Do NOT create a separate `monitoring/handlers.py` file (the research suggested it, but adding a new file for shared functions within the same module is over-engineering). Keep the shared functions in `dashboard.py` as private functions.
